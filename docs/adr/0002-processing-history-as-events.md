# Processing history is events

Academic Reader records Processing Events as the source of truth for a Processing Run and derives current progress from them. Events should emphasize worker/model activity, warnings, failures, degraded behavior, and long-running progress rather than every sub-second local step. The Reader View becomes available as soon as Source View, Blocks, and Reader View data exist; narration, audio, and other enhancements continue as non-blocking events. Events are persisted in Convex and delivered to clients through Convex subscriptions rather than a separate app-owned live stream, and v1 avoids app-owned manual retry loops in workers or backend orchestration. Reader-facing progress may use compact read models maintained transactionally from inserted Processing Events; those projections are caches and must remain rebuildable from the Processing Event history. Fine-grained worker progress is a v1 requirement for each enabled Conversion Model and must be implemented case-by-case against real runs: Marker should use a small tqdm hook inside the worker that emits structured events, while LightOnOCR and Chandra need their own tested progress adapters based on the progress surfaces their libraries actually expose. A Conversion Model can remain configured but disabled until this progress support is real, so the app does not build downstream logic around too-coarse events. Each Processing Event includes a required emitter timestamp (`emittedAt`) from the worker or app process that produced it, but events are ordered and displayed in the main UI by Convex `_creationTime`; `emittedAt` is retained as debug evidence and v1 does not add an explicit sequence field. Event `type` is a strict namespaced union for enabled v1 event types, not an arbitrary string; adding a new event requires adding it to the shared event type definition and tests. Conversion events stay broad/common, while Narration uses explicit phase-specific event types because candidate extraction, eligibility review, guide generation, rewrite, and audio generation are fixed reader-visible phases. Each event has a required `emitter` (`app`, `marker`, `lightonocr`, `chandra`, `kokoro`, or `qwen3`) plus optional common fields such as `blockId`, `pageNumber`, `progress`, and `data` so worker-specific details do not force a separate table or premature payload taxonomy.
