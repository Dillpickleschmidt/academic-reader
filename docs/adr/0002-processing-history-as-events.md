# Processing history is append-only

Academic Reader records Processing Events as the source of truth for a Processing Run and derives current progress from them. Events should emphasize worker/model activity, warnings, failures, degraded behavior, and long-running progress rather than every sub-second local step. The Readable View becomes available as soon as Source View, Blocks, and Readable View data exist; narration, audio, and other enhancements continue as non-blocking events. Events are delivered by Server-Sent Events rather than polling, and v1 avoids app-owned manual retry loops in workers or backend orchestration. Fine-grained worker progress is a v1 requirement for each enabled Conversion Model and must be implemented case-by-case against real runs: Marker should use a small tqdm hook inside the worker that emits structured events, while LightOnOCR and Chandra need their own tested progress adapters based on the progress surfaces their libraries actually expose. A Conversion Model can remain configured but disabled until this progress support is real, so the app does not build downstream logic around too-coarse events. Each Processing Event includes a required emitter timestamp (`emittedAt`) from the worker or app process that produced it, but events are ordered and displayed in the main UI by Convex `_creationTime`; `emittedAt` is retained as debug evidence and v1 does not add an explicit sequence field. Event `type` is a strict broad/common namespaced union for enabled v1 event types, not an arbitrary string; adding a new event requires adding it to the shared event type definition and tests. Each event has a required `emitter` (`app`, `marker`, `lightonocr`, `chandra`, `kokoro`, or `qwen3`) plus optional common fields such as `blockId`, `pageNumber`, `progress`, and `data` so worker-specific details do not force a separate table or premature payload taxonomy.
